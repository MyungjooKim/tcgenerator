## [회원 가입 조건]

### 정책 목표
- 유스메타 링크를 통해 가입한 사용자에게만 특정 기능을 제공하여, 일반 가입자와 구분 관리한다.

### 핵심 정책 규칙
- 유스메타 링크를 통한 가입 여부를 식별하는 **별도 상태값(플래그)**을 회원 데이터에 추가한다.
- 이 상태값을 기반으로 회원 전용 기능(오더폼, 알림 등)의 활성화 여부를 제어한다.

### 예외/제한 사항
- 유스메타 링크 미경유 가입자는 관련 기능 접근 불가 (또는 비활성화 노출)

### 미결 사항
- 상태값 컬럼명 및 값 정의 필요 (예: `is_yoosemeta: boolean`)
- 기존 가입자 마이그레이션 처리 방침 미정

## [오더폼 구성]

### 정책 목표
- 유스메타 회원에게 제공하는 오더폼 내 레버리지 변경 영역을 서비스 정책에 맞게 제어한다.

### 핵심 정책 규칙
- 오더폼 내 레버리지 변경 UI는 서비스 허용 범위(1배, 2배)에 맞게 제한된다.
- 2배 초과 레버리지 상태인 경우, 오더폼 내 해당 영역을 **비활성화** 처리한다.

### 예외/제한 사항
- 거래소에서 직접 레버리지를 올린 경우(예: Bybit 10배 세팅), 우리 화면에서 조회는 가능하나 수정은 2배 초과 불가

### 미결 사항
- 비활성화 시 사용자에게 안내 문구 노출 여부 및 문구 내용 미정

## [알림 기능]

### 정책 목표
- 유스메타 회원에게 거래 관련 알림을 다양한 채널(채팅방, 메신저, 앱 내)을 통해 제공한다.

### 핵심 정책 규칙
- **오픈 채팅방**: 회원별 개별 오픈 채팅방을 생성하여 운영한다.
- **외부 메신저 알림**: 카카오톡, 텔레그램을 통해 알림을 발송한다.
- **긴급 Close 기능**: 모바일 웹에서 로그인 후 긴급하게 포지션을 Close 할 수 있는 기능을 제공한다.
- 알림 대상: 회원 전용 (비회원 또는 유스메타 비연동 회원 제외)

### 예외/제한 사항
- 긴급 Close는 모바일 웹 로그인 상태에서만 가능
- 외부 메신저 알림은 수신 동의한 회원에 한함 (정책 확인 필요)

### 미결 사항
- 오픈 채팅방 플랫폼 미정 (카카오 오픈채팅 vs 자체 채팅)
- 텔레그램 봇 연동 방식 미정
- 알림 발송 트리거 조건 목록 미정 (어떤 이벤트에 알림을 보낼지)
- 긴급 Close의 대상 포지션 범위 (전체 / 선택)

## [레버리지 2배 제한]

### 정책 목표
- 서비스 내 모든 주문에서 레버리지를 최대 2배로 제한하여 과도한 위험을 방지한다.

### 핵심 정책 규칙

**레버리지 제한 범위**
- 제공 레버리지: 1배, 2배만 허용
- 초기 주문, 추가 주문, 변경 주문 모두 동일하게 적용

**Available Balance(AB) 처리**
- 레버리지 변경 시 AB 값이 변동되나, 현재 변경 허용 여부 불명확 → 미결
- 1배 → 2배: AB 변경됨 (검토 필요)
- 2배 → 1배: AB 변경됨 (검토 필요)

**레버리지 변경 제한 조건**
- 선택된 코인에 Open Position 또는 Open Order가 존재하는 경우 레버리지 변경 불가

**자동 동기화 (서버)**
- 서버에서 주기적으로 레버리지를 자동 동기화
- 거래소 레버리지 상태 기준으로 1배 / 2배 / 2.1배 이상 세 가지 케이스 분기 처리

**거래소 초기 세팅 처리**
- Bybit: 기본 10배 세팅 → 서비스 초기화 시 2배로 일괄 변경 (정책 확정 필요)

**UI 토스트 메시지**
- 레버리지 변경 성공 시: "성공적으로 변경되었습니다." 안내 토스트 노출
- 레버리지 변경 실패 시: 토스트 미노출 (잦은 재시도 케이스)

**화면 자동 업데이트**
- 해당 창이 포커스 상태인 경우에만 자동 업데이트
- 창이 2개 이상일 경우, 포커스된 창 1개만 업데이트 (Visible이어도 포커스 없으면 제외)

**멀티 디바이스/브라우저**
- 다른 PC 또는 별도 브라우저에서 거래 시 동작 검증 필요

### 예외/제한 사항
- 2배 초과 레버리지(예: Bybit 10배)로 세팅된 경우, 화면 표시 방법 별도 고려 필요
- 변경 실패 케이스 중 잦은 재시도(네트워크 이슈 등)는 토스트 미노출로 처리

### 미결 사항
- AB(Available Balance) 변경 허용 여부 및 처리 방식
- 2배 초과 레버리지 상태 화면 표시 UI/UX 방안
- Bybit 일괄 변경 자동화 범위 및 시점
- 서버 자동 동기화 주기 미정
- 바이낸스 정합 (별도 2주 정합 기간 운영)

## [자동 TP/SL 주문]

### 정책 목표
- 주문 시 자동으로 TP(Take Profit)/SL(Stop Loss)를 설정하여, 사용자가 별도 설정 없이도 리스크를 관리할 수 있게 한다.

### 핵심 정책 규칙
- 자동 TP/SL 기능은 필수 제공 기능이다.
- **설정 옵션**: 사용자 설정에서 ON/OFF 제공
- **기본값**: ±10% (파트너 계정과 개인 계정의 기본값을 다르게 설정 가능)
- 추가 주문(add) 시마다 TP/SL이 각각 삽입되어 Partial TP/SL이 여러 개 생길 수 있음

### 예외/제한 사항
- Gate 거래소: 현재 Basic TP/SL 미지원으로 해당 기능 적용 불가 (개발 복잡도 높음)
  - Gate 제외 거래소 우선 적용 후 별도 검토
- 추가 주문마다 생성되는 Partial TP/SL 누적 처리 방식 별도 정의 필요

### 미결 사항
- Gate 거래소 대응 시점 및 방식
- 파트너/개인 계정별 기본값 수치 확정
- Partial TP/SL 누적 시 최대 허용 개수 또는 병합 정책

## [기타 참고 사항]

### 사용자 특성
- 주 사용자층: 50대 이상, IT 친숙도가 낮을 수 있음
- UI/UX 설계 시 단순하고 명확한 안내 문구 사용 필요

### 거래소 정합
- 바이낸스: 2주 정합 기간 운영 후 서비스 적용
