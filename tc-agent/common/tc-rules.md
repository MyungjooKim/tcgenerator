# TC 작성 규칙 — Supercycl 공통

> **이 파일은 모든 Phase에 공통으로 적용되는 규칙이다.**
> tc-writer는 작성 전 반드시 이 파일을 먼저 읽고, 이어서 해당 Phase의 `tc-rules-override.md`를 읽어 덮어씌운다.
> Phase 전용 규칙이 이 파일의 규칙보다 우선한다.

---

## 1. 기본 원칙

- 모든 테스트는 **사람이 직접 수행**하는 수동 테스트를 기준으로 작성한다.
- 테스트 단계는 **비개발자도 따라할 수 있을 정도**로 구체적으로 작성한다.
- 모호한 표현 ("제대로 동작한다", "정상적으로") 대신 **측정 가능한 결과**를 명시한다.
- 하나의 TC는 **하나의 검증 가능한 단위 행동**만 다룬다.

## 2. TC ID 규칙

### 2-1. 신규 형식

형식: `{ProjectCode}-{SuiteCode}-{NNN}`
예시: `SC-GNBF-001`, `SC-TRD-ACCT-001`, `SC-TRD-ORDR-012`

- SuiteCode = 시트명의 약어 (예: GNB&Footer → GNBF, Trade-Account Info → TRD-ACCT)
- 같은 시트 내에서도 거래소 분기 등에 따라 SuiteCode가 달라질 수 있음 (예: TRD-ACCT vs TRD-ACCT2)

| 요소 | 규칙 | 예시 |
|------|------|------|
| ProjectCode | 프로젝트 코드 | `SC` (PC Web), `SM` (Mobile Web) |
| SuiteCode | 시트명 약어 (대문자, 하이픈 허용) | `GNBF`, `TRD-ACCT`, `LOGN` |
| NNN | Suite 내 001부터 순차 번호 | `001`, `012` |

### 2-2. 구형 형식 (분류표 없는 경우 폴백)

형식: `[도메인코드]-[기능코드]-[번호]`
예시: `LEVR-POPUP-001`

> classification_v1_APPROVED.md가 없는 Phase에서만 사용.

### 도메인 코드 고정 목록 (임의 추가 금지)

| 코드 | 한글명 | 포함 범위 |
|------|--------|----------|
| AUTH | 인증/온보딩 | 로그인, 약관, 온보딩 |
| REFF | 레퍼럴 | QR 스캔, partner_code, 공유 |
| LEVR | 레버리지 | 레버리지 설정, 안내 팝업, 제한 |
| TPSL | TP/SL | Auto TP/SL 설정, 주문 연동, 사후 변경 |
| TRAD | 트레이딩 | 주문(Market/Limit/Conditional), 포지션, 대시보드 |
| SGNL | 시그널 | 시그널 카드, 팔로우, 목록 |
| FUND | 자금 | 지갑, 잔고 표시, 자금 자동 전송 |
| MOBL | 모바일UI | 반응형 레이아웃, BottomNav, TESTNET 배지 |
| SETG | 설정 | 로그아웃, 계정 설정 |
| ROUT | 라우팅 | 접근 제어, 리다이렉트, 라우팅 시퀀스 |

> 새 도메인이 필요하면 이 파일에 추가하고 팀 협의 후 확정한다.

### 기능코드 규칙
- 대문자 영문 4~6자
- 번호: 001부터, **Positive → Negative → Edge 순서**

## 3. TC 분해 단위 기준

### 3-1. 분리해야 하는 경우
- **사전 조건이 다른 경우**
- THEN의 검증 포인트가 2개 이상이고 **각각 독립적으로 실패할 수 있는 경우**
- 시간 순서가 다른 세션에 걸친 경우 (예: 버튼 클릭 / 재방문 시 상태 유지)

### 3-2. 합쳐도 되는 경우
- 앞 단계 없이 뒤 단계를 시작할 수 없는 연속 흐름
- THEN이 동일한 행동의 즉각적인 결과만 확인하는 경우

### 3-3. 도메인별 TC 수량 가이드

| 도메인 기능 수 | 권장 TC 수 |
|--------------|-----------|
| 1~3개 기능   | 기능 수 × 2~3 |
| 4~6개 기능   | 기능 수 × 2 내외 |
| 7개 이상     | 기능 수 × 1.5 이하 |

## 4. THEN(예상 결과) 작성 깊이 기준

**기본 원칙: 테스터가 화면에서 직접 확인 가능한 수준**

**허용:**
```
- "I Understand" 버튼이 표시된다                  ← 실제 UI 텍스트
- 슬라이더 범위가 1~2로 제한되어 3x 이상 선택 불가  ← 화면에서 확인
- TradingPage 재진입 시 팝업이 표시되지 않는다      ← 행동으로 확인
```

**불허:**
```
- hasSeenLeverageNotice=true로 변경된다    ← 코드 내부 변수명
- DISMISS_LEVERAGE_NOTICE가 디스패치된다   ← Redux 액션명
```

> 코드 레벨 검증이 필요하면 비고란에 `[개발 확인]` 태그로 별도 명시.

### 오류 메시지 텍스트 처리

| 상황 | 작성 방법 |
|------|----------|
| 기획서/코드에 실제 텍스트 명시 | 그대로 사용 |
| 텍스트 미명시 | `"오류 메시지가 표시된다"` + 비고에 `[미결]` |

## 5. TC 생성 카테고리 (4가지)

각 도메인(중분류)에 대해 아래 4가지 카테고리 순서로 TC를 작성한다.
해당 카테고리에 만들 TC가 없으면 skip한다.

| # | 카테고리 | 분류 | 우선순위 | 설명 |
|---|---------|------|---------|------|
| 1 | **UI/UX 체크** | Positive | Medium~High | 화면 표시, 초기 상태, 레이아웃, 로딩 상태, 빈 데이터 표시 |
| 2 | **주요 기능** | Positive | High | 핵심 정상 흐름(Happy Path), CRUD, 상태 전환, 네비게이션 |
| 3 | **예외 기능** | Negative | High~Medium | 비즈니스 위험 시나리오 — 잘못된 입력, 권한 오류, 중복 요청, 잔고 부족 |
| 4 | **에러 처리 및 비기능** | Edge | Medium~Low | 네트워크 오류, 타임아웃, 빈 응답 방어, 성능, 접근성 |

### TC 번호 순서
같은 중분류 내에서 카테고리 1 → 2 → 3 → 4 순으로 번호를 매긴다.

## 5-1. Smoke TC 선별 기준

### 목적
새 빌드 배포 직후 핵심 기능이 동작하는지 빠르게 확인하는 TC 세트.
전체의 **20~30%** 를 목표로 선별한다.

### 선별 로직 (우선순위 순)

| 순서 | 조건 | 개수 |
|------|------|------|
| 1 | 중분류별 대표 Positive 1개 (High > Medium > Low 순) | 중분류당 1개 |
| 2 | 중분류별 High Negative 1개 (있을 때만) | 중분류당 최대 1개 |
| 3 | 대분류별 최소 1개 보장 | 위 조건으로 0개인 대분류만 |

### bold 처리
Smoke 대상 TC는 헤딩을 bold로 작성한다: `### **SC-{SuiteCode}-NNN** — 제목`

### 분류 값과 Smoke 관계
| 분류 | Smoke 대상 |
|------|-----------|
| Positive | 중분류당 1개 (카테고리 1, 2에서 선별) |
| Negative | High일 때만 중분류당 1개 (카테고리 3에서 선별) |
| Edge | Smoke 대상 아님 |

## 6. 거래소별 N/A 판정 (공통 프레임워크)

### 6-1. N/A 판정 기준

TC가 특정 거래소에 적용되지 않는 경우 해당 거래소 판정 컬럼을 **N/A**로 사전 기입한다.

| 상황 | N/A 적용 |
|------|---------|
| 거래소가 해당 기능 자체를 미지원 | 전체 Phase 해당 TC → N/A |
| 특정 UI 요소가 없음 (버튼, 컬럼 등) | 해당 TC → N/A |
| 이 TC가 다른 거래소를 대상으로 작성됨 | 나머지 거래소 → N/A |

### 6-2. TC 비고란 N/A 명시

N/A 적용 시 비고란에 이유를 기재한다:
```
**비고**
- Gate: Auto TP/SL 미지원 (N/A)
- Bitget/Hyperliquid: Open Orders 수정 버튼 없음 (N/A)
```

### 6-3. Phase별 거래소 목록

거래소 목록과 N/A 세부 규칙은 각 Phase의 `tc-rules-override.md`에 정의한다.
상세 거래소 특성은 `common/policy/01_TC_Policy_Exchange.md` 참조.

## 7. 테스터 배정 (공통 프레임워크)

테스터 배정은 각 Phase의 `tc-rules-override.md`에 정의한다.

**기본 배정 원칙:**
- 거래소 컬럼이 곧 판정 컬럼. 테스터는 담당 거래소 컬럼에 직접 Pass/Fail/N/T 기입.
- 별도 테스터 컬럼 없음. Excel Row 2(서브헤더)에 테스터 배정이 색상으로 표시.

**판정 코드:**

| 코드 | 의미 |
|------|------|
| Pass | 기대 결과와 일치 |
| Fail | 버그 발생 → 리포트 필수 |
| N/T | Not Tested (기본 초기값) |
| N/A | Not Applicable (거래소 미지원 등) |

## 8. TC 버전 히스토리

### 8-1. 파일명 규칙

```
SPCY_TC_{PhaseCode}_{YYYYMMDD}_v{N}.xlsx
```

| 요소 | 설명 | 예시 |
|------|------|------|
| PhaseCode | Phase 식별자 | P1_Youthmeta, P2_Mobile |
| YYYYMMDD | 빌드 날짜 | 20260402 |
| v{N} | 해당 날짜 내 버전 (1부터) | v1, v2 |

**예시:**
```
SPCY_TC_P1_Youthmeta_20260323_v1.xlsx
SPCY_TC_P2_Mobile_20260402_v1.xlsx
SPCY_TC_P2_Mobile_20260402_v2.xlsx   ← 같은 날 재빌드
```

### 8-2. 버전 증가 규칙
- 같은 날짜에 동일 Phase 파일이 이미 있으면 v{N}을 1씩 증가
- 날짜가 바뀌면 v1부터 다시 시작
- output-builder가 자동으로 버전 번호를 탐색하여 다음 번호를 사용

## 9-1. 사전 조건 작성 규칙

### 형식

```
**사전 조건**
1. [조건 A]
2. [조건 B]
3. [조건 C]
```

### 원칙

1. **개조식, 번호 필수** — 서술문 금지. 번호를 붙여 조건을 나열한다.
2. **설정 순서대로** — 테스터가 실제로 상태를 만드는 순서대로 작성한다.
   - 먼저 선행 계정/로그인 상태 → 화면/탭 진입 → 특정 UI 조작 완료 순
3. **명사형 또는 동사 완료형으로 끝맺음** — "이다" 제거. 아래 종결 패턴 사용:
   - `~인 상태` — 현재 상태 표현 (예: "Trade 메뉴인 상태")
   - `~된 상태` — 완료된 상태 (예: "로그인된 상태", "Limit 주문이 등록된 상태")
   - `~한 상태` — 동작 완료 (예: "약관에 동의한 상태", "Market 주문을 실행한 상태")
   - `~선택된 상태` — 선택 완료 (예: "Open Orders 탭이 선택된 상태")

### 허용 / 불허

| 구분 | 예시 |
|------|------|
| ✅ 허용 | `1. Trade 메뉴인 상태` |
| ✅ 허용 | `2. Open Order가 체크된 상태` |
| ✅ 허용 | `3. Limit Open Order 주문을 실행한 상태` |
| ✅ 허용 | `4. Open Orders 탭이 선택된 상태` |
| ❌ 불허 | `Google OAuth 로그인이 완료된 상태이다` (서술형 + "이다") |
| ❌ 불허 | `충분한 잔고가 있다` (번호 없음 + "있다" 종결) |
| ❌ 불허 | `Trade 화면에 접속해 있다` ("있다" 종결) |

### 추가 주석

Edge 조건이나 주의사항은 해당 항목 뒤에 `(* ...)` 형태로 붙인다:

```
3. 주문 수량이 부분 체결된 상태
   (* 거래량이 낮은 코인에서 체결될 가능성이 높은 호가의 size 이상 주문 후 대기하면 부분 체결됨)
```

## 12. 플랫폼별 테스트 기준

> tc-writer는 기능 TC 외에 아래 플랫폼 특화 체크리스트를 확인하고, 해당하는 TC를 MOBL 도메인 또는 관련 도메인에 추가한다.
> Phase의 플랫폼 구성은 `tc-rules-override.md`에서 확인한다.

### 12-1. 모바일 웹 공통 (Web(Mobile))

모바일 브라우저 기반 서비스라면 아래 항목을 반드시 검토한다.

| 체크 항목 | 검증 내용 | TC 추가 조건 |
|----------|----------|------------|
| **고정폭 레이아웃** | 360px(또는 Phase별 기준폭)에서 요소 잘림·겹침·가로 스크롤 없음 | 항상 추가 |
| **터치 영역** | 주요 버튼의 탭 영역이 충분히 크고, 인접 버튼 오탭 없음 | 버튼 밀집 화면에 추가 |
| **소프트 키보드** | 입력 필드 포커스 시 키보드 팝업 → 필드가 키보드에 가려지지 않음 | 입력 필드 있는 화면에 추가 |
| **바텀시트/모달** | 바텀시트·모달이 화면 전체를 덮지 않고 스크롤 가능 | 바텀시트 사용 화면에 추가 |
| **BottomNav 고정** | 스크롤 시 BottomNav가 화면 하단에 고정 유지됨 | BottomNav 있는 Phase에 추가 |
| **가로 모드** | landscape 회전 시 레이아웃 무너짐 없음 | 필요 시 추가 (Phase별 판단) |

### 12-2. iOS Safari 특화

iOS Safari에서만 발생하는 동작을 검증한다.

| 체크 항목 | 검증 내용 | 플랫폼 표기 |
|----------|----------|-----------|
| **주소창 스크롤 숨김** | 스크롤 시 Safari 주소창이 숨겨지면서 viewport 높이 변화 → BottomNav 위치 밀림 없음 | `iOS Safari` |
| **Safe Area (홈 바)** | 홈 바 영역에 BottomNav·버튼이 겹치지 않음 | `iOS Safari` |
| **노치 영역** | 노치 기기에서 헤더 콘텐츠가 잘리지 않음 | `iOS Safari` |
| **모멘텀 스크롤** | 빠른 스크롤(rubber band) 후 오버스크롤 시 UI 깨짐 없음 | `iOS Safari` |
| **키보드 팝업 처리** | iOS 키보드 팝업 시 화면이 위로 밀리거나 BottomNav가 키보드 위로 올라오지 않음 | `iOS Safari` |

### 12-3. Android Chrome 특화

Android Chrome에서만 발생하는 동작을 검증한다.

| 체크 항목 | 검증 내용 | 플랫폼 표기 |
|----------|----------|-----------|
| **뒤로가기 제스처** | Android 뒤로가기(스와이프 또는 시스템 버튼) 시 의도한 화면으로 이동 | `Android Chrome` |
| **주소창 크기 변화** | 스크롤 중 Chrome 주소창 축소·확대 시 레이아웃 재조정 정상 | `Android Chrome` |
| **키보드 팝업 처리** | Android 키보드 팝업 시 화면 리사이즈 발생 → 레이아웃 깨짐 없음 | `Android Chrome` |

### 12-4. 플랫폼별 TC 작성 원칙

1. **공통 기능 TC**는 플랫폼 필드를 `Web(Mobile)`로 표기한다.
2. **iOS 전용 동작**은 동일 TC를 `iOS Safari`로 표기하거나 별도 TC를 추가한다.
3. **Android 전용 동작**은 `Android Chrome`으로 표기한다.
4. **플랫폼 무관 로직**(서버 응답, 데이터 계산)은 `공통`으로 표기한다.
5. 플랫폼 특화 TC는 **MOBL 도메인**에 배치한다. 연관 기능에 직접 붙이는 것도 허용한다.
6. 최소 TC 세트에는 각 플랫폼별 대표 케이스를 1개 이상 포함한다.

### 12-5. 웹(Desktop) 서비스 추가 시 체크리스트

향후 데스크톱 웹 Phase가 추가될 경우 아래 항목을 검토한다.

| 체크 항목 | 검증 내용 |
|----------|----------|
| **반응형 브레이크포인트** | 768px / 1024px / 1280px 등 기준 해상도에서 레이아웃 확인 |
| **마우스 호버 상태** | hover 시 버튼·링크 색상·커서 변화 확인 |
| **키보드 단축키** | Tab 이동, Enter 실행, Esc 닫기 등 키보드 접근성 |
| **브라우저 호환** | Chrome / Safari / Firefox / Edge 별 동작 차이 |
| **우클릭·드래그** | 의도치 않은 컨텍스트 메뉴, 드래그 셀렉션 처리 |
