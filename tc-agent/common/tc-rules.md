# TC 작성 규칙 — Supercycl 공통

> **이 파일은 모든 Phase에 공통으로 적용되는 규칙이다.**
> tc-writer는 작성 전 반드시 이 파일을 먼저 읽고, 이어서 해당 Phase의 `tc-rules-override.md`를 읽어 덮어씌운다.
> Phase 전용 규칙이 이 파일의 규칙보다 우선한다.

---

## 1. 기본 원칙

- 모든 테스트는 **사람이 직접 수행**하는 수동 테스트를 기준으로 작성한다.
- 테스트 단계는 **비개발자도 따라할 수 있을 정도**로 구체적으로 작성한다.
- 모호한 표현 ("제대로 동작한다", "정상적으로") 대신 **측정 가능한 결과**를 명시한다.
- 하나의 TC는 **하나의 검증 가능한 단위 행동**만 다룬다.

## 1-1. 스펙 기반 생성 원칙 (최우선 — 모든 규칙에 선행)

### 원칙 A. 추측 금지 (No-Speculation)

기획서(스펙)에 **명시되지 않은 시나리오는 TC로 생성하지 않는다**. 특히 아래 항목은 스펙에 구체적 명시가 없으면 TC 생성 금지:

| 금지 유형 | 예시 | 처리 방법 |
|---------|------|----------|
| 오프라인/네트워크 끊김 동작 | "비행기 모드에서 Splash 접근" | 스펙에 명시 없으면 `[미결]` 태그로 비고 기록 |
| 성능 임계치 | "3초 이내 로딩", "15초 타임아웃" | 스펙의 숫자만 사용. 없으면 TC 생성 금지 |
| 특정 상태에서의 리다이렉트 | "로그인 완료 상태에서 Splash URL 직접 접근" | 스펙에 라우팅 정책 명시된 경우만 |
| 재시도/쿨다운 | "30초 쿨다운" | 스펙에 수치 명시된 경우만 |

`[미결]` 태그로 처리한 항목은 PM/기획자 확인이 필요한 TBD 목록이다.

### 원칙 B. 에러 케이스 테이블 1:1 매핑

기획서에 다음 패턴의 표가 있으면:

```markdown
**에러 케이스**:
| 에러 케이스 | 표시 패턴 | 메시지 | 동작 |
|-----------|---------|-------|------|
| 잘못된 인증 코드 | Inline Error | "Incorrect code..." | 입력 초기화 |
| 인증 코드 만료 | Modal | "Code expired..." | Resend/Cancel |
```

→ 각 행은 **정확히 1개의 Negative TC**로 변환한다. 더 쪼개지 말고, 합치지도 말 것.
→ 에러 메시지는 스펙의 원문 그대로 사용한다.
→ 스펙에 에러 테이블이 없는 화면은 **Negative TC를 만들지 않는다** (억지로 만들지 말 것).

### 원칙 C. Deferred / Archived 화면 제외

스펙의 화면 목록에서 `상태 (status)` 컬럼이 다음인 화면은 TC 생성 대상에서 **완전히 제외**한다:

| 상태 값 | 의미 | 처리 |
|--------|------|------|
| `active` | 정식 화면 | TC 생성 |
| `deferred` | 다른 화면의 상태 변형 (별도 목업 없음) | **제외** — 원본 화면 TC에 상태 변형을 병합 |
| `archived` | 폐기 / 사용 안함 | **제외** |

### 원칙 D. Visual 속성 통합 (Test Pyramid)

색상·폰트·radius·여백 등 **시각 속성은 화면당 1개의 "UI Spec Compliance" TC**로 통합한다. 분할 생성 금지.

| ❌ 금지 (분할) | ✅ 권장 (통합) |
|--------------|--------------|
| TC-001: 배경색 #000000 확인 | TC-001: **디자인 토큰 준수** — 배경(#000000), 슬로건 색상(Supercycl #00E676), PrimaryButton 스타일(녹색/흰 텍스트/radius 8px) 일괄 확인 |
| TC-002: 슬로건 텍스트 확인 | |
| TC-003: 버튼 색·너비·radius 확인 | |

이유: 시각 속성 검증은 Visual Regression 도구(Chromatic/Percy) 영역이며, 수동 TC에서 픽셀 단위로 쪼개면 유지보수성이 붕괴된다.

### 원칙 E. 비기능 TC는 **화면 성격 기반 조건부** 생성 (원칙 A의 예외)

원칙 A(추측 금지)에도 불구하고, **화면 성격상 필연적으로 검증해야 하는 비기능 항목**은 TC로 생성한다. 단, 스펙에 수치 명시가 없으면 업계 표준 수치를 사용하고 `[미결]` 태그로 "임계치 PM 확정 필요" 명시.

#### 화면 성격 6분류 → 비기능 TC 매트릭스

| 성격 (character) | 예시 화면 | 필수 비기능 TC | 업계 표준 임계치 |
|-----------------|----------|--------------|---------------|
| **entry** (진입) | Splash, Landing, Login Options | **로딩 시간** | 3초 (Google RAIL, Apple HIG) |
| **data-fetch** (데이터 조회) | Markets, Portfolio, Order History | **초기 로딩 시간** + 빈 상태 표시 + 새로고침 | 5초 (API + 렌더) |
| **realtime** (실시간) | Trade Lite, Positions | **WebSocket 끊김 복원** + 지연 표시 배너 | 재연결 10초 이내 |
| **form** (입력) | Email Input, Amount 입력, Limit Price | **입력 반응 지연** (선택) | 100ms (RAIL first-input-delay) |
| **overlay** (시트/모달) | Coin Selector, Margin Mode Sheet, Cancel Order Modal | **외부 탭 닫힘** + 진입/퇴장 애니메이션 | - |
| **static** (정적) | Terms of Use, Privacy Policy, Guide | 비기능 TC 불필요 (스크롤만) | - |

#### 기존 원칙과의 우선순위

- 원칙 B(에러 테이블 1:1)가 최우선 → 스펙에 로딩 실패 에러가 명시되어 있으면 그걸 따름
- 원칙 A(추측 금지)는 **임계치 수치 자체에만** 적용 → 업계 표준을 쓰더라도 `[미결]` 태그 필수
- 성격이 `static`인 화면은 비기능 TC 생성 금지

#### 예시 — Splash(entry) 로딩 시간 TC

```
### SM-SPL-004 — Splash 화면 로딩 시간

| 분류 | Edge |
| 우선순위 | Medium |

**사전 조건**
1. 3G 또는 저사양 기기 (Moto G4 등) 환경인 상태

**테스트 단계**
앱 URL 접속 후 Splash 화면의 슬로건과 Get Started 버튼이
모두 표시되는 시점까지의 시간을 측정한다.

**예상 결과**
- Splash 완전 표시까지 3초 이내 (업계 표준, Google RAIL 기준)
- 슬로건·로고·버튼이 깨지거나 잘리지 않는다

**비고**
- [미결] 3초 임계치는 업계 표준 기반 추정. PM/디자이너 확정 필요.
```

### 원칙 F. 네비게이션 동작은 **화면 분류 기반 조건부** TC 생성

화면은 뒤로가기/스와이프 시 동작이 서로 다르다. `screen_code_map.md`의 **`네비` 컬럼** 기준으로 필수 네비게이션 TC를 추가한다. 성격(character)과 네비게이션은 **직교하는 축** — 예: Splash/Login Complete 모두 `entry` 성격이지만 네비게이션은 다름.

#### 네비게이션 6분류 → 필수 TC

| 네비 (navigation) | 기대 동작 | 필수 TC |
|------------------|----------|--------|
| **tab-root** | 앱 종료 확인 또는 Splash 복귀 | 탭 최상위에서 뒤로가기 시 종료 처리 |
| **one-way** | **이전 화면 복귀 금지** (완료 화면) | 뒤로가기/스와이프 **차단 또는 홈 이동** — 인증/주문 재실행 방지 |
| **sequential** | 이전 단계 복귀 + 입력값 유지 | 뒤로가기 → 이전 단계 + 입력값 preserved |
| **overlay** | 오버레이만 닫힘, 배경 유지 | 뒤로가기 / 외부 탭 → 모달만 dismiss |
| **detail** | 리스트 복귀 + 스크롤 위치 | 뒤로가기 → 리스트 + scroll restore |
| **static** | 자연스러운 상위 복귀 | 기본 네비 (별도 TC 불필요, 단 depth 2 이상이면 상위 화면 확인) |

#### one-way 예시 — Login Complete 뒤로가기 차단 TC (사용자 지적 케이스)

```
### SM-LGC-XXX — 로그인 완료 후 뒤로가기 차단

| 분류 | Negative |
| 우선순위 | High |

**사전 조건**
1. 모바일 브라우저에서 접속한 상태
2. 이메일 인증이 완료되어 Login Complete 화면(SCR-009)에 진입한 상태

**테스트 단계**
브라우저 뒤로가기 버튼 또는 iOS 스와이프 백 제스처를 실행한다.

**예상 결과**
- Verification Code(SCR-005) 또는 Email Input(SCR-003) 화면으로 돌아가지 않는다
- 다음 중 하나의 동작이 발생한다:
  a) 홈 화면(Markets/Trade Lite)으로 이동
  b) 뒤로가기가 차단되어 Login Complete 화면 유지
  c) "로그아웃하시겠습니까?" 확인 모달 표시
- 이미 완료된 인증 플로우를 재실행할 수 없는 상태가 유지된다

**비고**
- [미결] 구체적 차단 방식(이동/차단/모달)은 PM 확정 필요
- 일반적으로 `history.replaceState()` + popstate 핸들러로 구현
```

#### 기본 추론 (navigation 컬럼 비어있을 때)

| character | 기본 navigation |
|-----------|-----------------|
| overlay | overlay |
| static | static |
| 그 외 | sequential (안전한 기본값) |

명시적 `one-way`, `tab-root`, `detail` 지정은 **사람이 직접** 매핑해야 한다 (자동 추론 금지 — 잘못된 차단 로직은 UX 훼손).

## 1-2. 리스크 기반 TC 수량 (Risk-Based Quota)

### 화면별 리스크 스코어 산출

```
Risk Score = Impact × (UI 컴포넌트 수 + 인터랙션 수 + 에러 케이스 수)

Impact 가중치:
  - 자산/거래 (Trade/Order/Position)      = 5
  - 보안/인증 (Login/Auth/OAuth)          = 4
  - 데이터 (Portfolio/History)            = 3
  - 설정/프로필 (Settings/Profile)         = 2
  - 브랜딩/안내 (Splash/Guide/Landing)    = 1
```

### 권장 TC 수 (화면당)

| Impact | 스펙 에러 케이스 수 | 권장 TC 수 |
|--------|------------------|-----------|
| 1 (브랜딩) | 0~1 | **3~4개** |
| 2 (설정) | 1~3 | 5~8개 |
| 3 (데이터) | 2~5 | 7~12개 |
| 4 (보안) | 3~7 | 10~15개 |
| 5 (거래) | 5~10+ | 15~20개 |

### 강제 규칙

1. **스펙에 에러 케이스 0건인 화면은 Negative TC 0개**. Positive + 최소 Visual 1개로 구성.
2. **Splash·Landing 같은 단순 화면에 10개 이상 TC 생성 금지**. 대부분 3~4개로 충분.
3. 카테고리 비율(1:2:3:4 = 20:35:25:20)은 리스크가 높은 화면에만 적용. 단순 화면은 비율보다 **총량 제한**을 우선.

## 2. TC ID 규칙

### 2-1. 신규 형식

형식: `{ProjectCode}-{SuiteCode}-{NNN}`
예시: `SC-GNBF-001`, `SC-TRD-ACCT-001`, `SC-TRD-ORDR-012`

- SuiteCode = 시트명의 약어 (예: GNB&Footer → GNBF, Trade-Account Info → TRD-ACCT)
- 같은 시트 내에서도 거래소 분기 등에 따라 SuiteCode가 달라질 수 있음 (예: TRD-ACCT vs TRD-ACCT2)

| 요소 | 규칙 | 예시 |
|------|------|------|
| ProjectCode | 프로젝트 코드 | `SC` (PC Web), `SM` (Mobile Web), `SA` (Mobile App 향후) |
| SuiteCode | 시트명 약어 (대문자, 하이픈 허용) | `GNBF`, `TRD-ACCT`, `LOGN` |
| NNN | **Suite 내에서 전역 유니크**한 순차 번호 | `001`, `012` |

### ⚠️ SeqNumber 유니크 규칙 (중요)

TC ID는 **전역 고유**해야 한다 (ISO/IEC/IEEE 29119-3 요구사항). 프로젝트 유형에 따라 유니크 스코프가 다르다:

#### 유형 A. Suite-based (PC Web 기본)

- **NNN은 Suite 내에서 전역 유니크**. 중분류가 바뀌어도 리셋 금지.
- 예: `SC-GNBF-001` ~ `SC-GNBF-050` (GNB&Footer Suite 전체 연속)
- 사용처: PC Web (`SC`), 기능 중심 분류를 사용하는 프로젝트

#### 유형 B. Screen-based (Mobile 권장)

- **SuiteCode 자리에 ScreenCode**를 사용. **NNN은 화면(중분류) 내에서 유니크**.
- 화면별 코드는 `projects/{project}/screen_code_map.md` 참조.
- 예:
  ```
  SM-SPL-001 ~ SM-SPL-004   (Splash 화면 내 001부터)
  SM-LGI-001 ~ SM-LGI-012   (Login Options 화면 내 001부터)
  SM-VCD-001 ~ SM-VCD-015   (Verification Code 화면 내 001부터)
  ```
- 사용처: Mobile (`SM`), 화면 수가 많고 스펙이 SCR-ID로 화면을 명시하는 프로젝트

#### 과거 버그 방지

| ❌ 금지 (리셋 버그) | ✅ 유형 A (Suite-based) | ✅ 유형 B (Screen-based) |
|-------------------|---------------------|---------------------|
| `SM-ONBD-001` (Splash) | `SM-ONBD-001` (Splash) | `SM-SPL-001` (Splash) |
| `SM-ONBD-001` (Login, 중복!) | `SM-ONBD-011` (Login) | `SM-LGI-001` (Login) |
| `SM-ONBD-001` (Email, 중복!) | `SM-ONBD-023` (Email) | `SM-EML-001` (Email) |
| `SM-ONBD-001` (Verify, 중복!) | `SM-ONBD-036` (Verify) | `SM-VCD-001` (Verify) |

프로젝트 유형은 프로젝트 정책 파일(`projects/{project}/0_tc_generation_rules.md`)에서 지정한다.

### 2-2. 구형 형식 (분류표 없는 경우 폴백)

형식: `[도메인코드]-[기능코드]-[번호]`
예시: `LEVR-POPUP-001`

> classification_v1_APPROVED.md가 없는 Phase에서만 사용.

### 도메인 코드 고정 목록 (임의 추가 금지)

| 코드 | 한글명 | 포함 범위 |
|------|--------|----------|
| AUTH | 인증/온보딩 | 로그인, 약관, 온보딩 |
| REFF | 레퍼럴 | QR 스캔, partner_code, 공유 |
| LEVR | 레버리지 | 레버리지 설정, 안내 팝업, 제한 |
| TPSL | TP/SL | Auto TP/SL 설정, 주문 연동, 사후 변경 |
| TRAD | 트레이딩 | 주문(Market/Limit/Conditional), 포지션, 대시보드 |
| SGNL | 시그널 | 시그널 카드, 팔로우, 목록 |
| FUND | 자금 | 지갑, 잔고 표시, 자금 자동 전송 |
| MOBL | 모바일UI | 반응형 레이아웃, BottomNav, TESTNET 배지 |
| SETG | 설정 | 로그아웃, 계정 설정 |
| ROUT | 라우팅 | 접근 제어, 리다이렉트, 라우팅 시퀀스 |

> 새 도메인이 필요하면 이 파일에 추가하고 팀 협의 후 확정한다.

### 기능코드 규칙
- 대문자 영문 4~6자
- 번호: 001부터, **Positive → Negative → Edge 순서**

## 3. TC 분해 단위 기준

### 3-1. 분리해야 하는 경우
- **사전 조건이 다른 경우**
- THEN의 검증 포인트가 2개 이상이고 **각각 독립적으로 실패할 수 있는 경우**
- 시간 순서가 다른 세션에 걸친 경우 (예: 버튼 클릭 / 재방문 시 상태 유지)

### 3-2. 합쳐도 되는 경우
- 앞 단계 없이 뒤 단계를 시작할 수 없는 연속 흐름
- THEN이 동일한 행동의 즉각적인 결과만 확인하는 경우

### 3-3. 도메인별 TC 수량 가이드 (폴백)

> ⚠️ **섹션 1-2 리스크 기반 쿼터가 이 규칙보다 우선한다.**
> 이 표는 스펙에 리스크 근거가 불명확할 때만 사용한다.

| 도메인 기능 수 | 권장 TC 수 |
|--------------|-----------|
| 1~3개 기능   | 기능 수 × 2~3 |
| 4~6개 기능   | 기능 수 × 2 내외 |
| 7개 이상     | 기능 수 × 1.5 이하 |

## 4. THEN(예상 결과) 작성 깊이 기준

**기본 원칙: 테스터가 화면에서 직접 확인 가능한 수준**

**허용:**
```
- "I Understand" 버튼이 표시된다                  ← 실제 UI 텍스트
- 슬라이더 범위가 1~2로 제한되어 3x 이상 선택 불가  ← 화면에서 확인
- TradingPage 재진입 시 팝업이 표시되지 않는다      ← 행동으로 확인
```

**불허:**
```
- hasSeenLeverageNotice=true로 변경된다    ← 코드 내부 변수명
- DISMISS_LEVERAGE_NOTICE가 디스패치된다   ← Redux 액션명
```

> 코드 레벨 검증이 필요하면 비고란에 `[개발 확인]` 태그로 별도 명시.

### 오류 메시지 텍스트 처리

| 상황 | 작성 방법 |
|------|----------|
| 기획서/코드에 실제 텍스트 명시 | 그대로 사용 |
| 텍스트 미명시 | `"오류 메시지가 표시된다"` + 비고에 `[미결]` |

## 5. TC 생성 카테고리 (4가지)

각 도메인(중분류)에 대해 아래 4가지 카테고리 순서로 TC를 작성한다.
해당 카테고리에 만들 TC가 없으면 skip한다.

| # | 카테고리 | 분류 | 우선순위 | 설명 |
|---|---------|------|---------|------|
| 1 | **UI/UX 체크** | Positive | Medium~High | 화면 표시, 초기 상태, 레이아웃, 로딩 상태, 빈 데이터 표시 |
| 2 | **주요 기능** | Positive | High | 핵심 정상 흐름(Happy Path), CRUD, 상태 전환, 네비게이션 |
| 3 | **예외 기능** | Negative | High~Medium | 비즈니스 위험 시나리오 — 잘못된 입력, 권한 오류, 중복 요청, 잔고 부족 |
| 4 | **에러 처리 및 비기능** | Edge | Medium~Low | 네트워크 오류, 타임아웃, 빈 응답 방어, 성능, 접근성 |

### ⚠️ 카테고리별 생성 제한 (섹션 1-1, 1-2와 연동)

- **카테고리 1 (UI/UX)**: 화면당 **1개의 "UI Spec Compliance" TC로 통합**. 색·폰트·radius를 분할 생성 금지 (원칙 D).
- **카테고리 3 (예외)**: **스펙의 에러 케이스 테이블에 있는 항목만** 생성. 추측 기반 Negative TC 금지 (원칙 A, B).
- **카테고리 4 (에러/비기능)**: 스펙에 수치(타임아웃 초/재시도 수)가 명시된 경우만 생성. 명시 없으면 `[미결]` 태그로 처리.
- **Impact 낮은 화면(Splash·Guide·About 등)**: 카테고리 3, 4는 대부분 skip. 전체 3~4개 TC 권장.

### TC 번호 순서
같은 Suite 내에서 **카테고리 1 → 2 → 3 → 4 순으로 NNN을 전역 연속 증가**시킨다.
중분류가 바뀌어도 NNN을 리셋하지 않는다 (섹션 2-1의 SeqNumber 전역 유니크 규칙).

## 5-1. Smoke TC 선별 기준

### 목적
새 빌드 배포 직후 핵심 기능이 동작하는지 빠르게 확인하는 TC 세트.
전체의 **20~30%** 를 목표로 선별한다.

### 선별 로직 (우선순위 순)

| 순서 | 조건 | 개수 |
|------|------|------|
| 1 | 중분류별 대표 Positive 1개 (High > Medium > Low 순) | 중분류당 1개 |
| 2 | 중분류별 High Negative 1개 (있을 때만) | 중분류당 최대 1개 |
| 3 | 대분류별 최소 1개 보장 | 위 조건으로 0개인 대분류만 |

### bold 처리
Smoke 대상 TC는 헤딩을 bold로 작성한다: `### **SC-{SuiteCode}-NNN** — 제목`

### 분류 값과 Smoke 관계
| 분류 | Smoke 대상 |
|------|-----------|
| Positive | 중분류당 1개 (카테고리 1, 2에서 선별) |
| Negative | High일 때만 중분류당 1개 (카테고리 3에서 선별) |
| Edge | Smoke 대상 아님 |

## 6. 거래소별 N/A 판정 (공통 프레임워크)

### 6-1. N/A 판정 기준

TC가 특정 거래소에 적용되지 않는 경우 해당 거래소 판정 컬럼을 **N/A**로 사전 기입한다.

| 상황 | N/A 적용 |
|------|---------|
| 거래소가 해당 기능 자체를 미지원 | 전체 Phase 해당 TC → N/A |
| 특정 UI 요소가 없음 (버튼, 컬럼 등) | 해당 TC → N/A |
| 이 TC가 다른 거래소를 대상으로 작성됨 | 나머지 거래소 → N/A |

### 6-2. TC 비고란 N/A 명시

N/A 적용 시 비고란에 이유를 기재한다:
```
**비고**
- Gate: Auto TP/SL 미지원 (N/A)
- Bitget/Hyperliquid: Open Orders 수정 버튼 없음 (N/A)
```

### 6-3. Phase별 거래소 목록

거래소 목록과 N/A 세부 규칙은 각 Phase의 `tc-rules-override.md`에 정의한다.
상세 거래소 특성은 `common/policy/01_TC_Policy_Exchange.md` 참조.

## 9-1. 사전 조건 작성 규칙

### 형식

```
**사전 조건**
1. [조건 A]
2. [조건 B]
3. [조건 C]
```

### 원칙

1. **개조식, 번호 필수** — 서술문 금지. 번호를 붙여 조건을 나열한다.
2. **설정 순서대로** — 테스터가 실제로 상태를 만드는 순서대로 작성한다.
   - 먼저 선행 계정/로그인 상태 → 화면/탭 진입 → 특정 UI 조작 완료 순
3. **명사형 또는 동사 완료형으로 끝맺음** — "이다" 제거. 아래 종결 패턴 사용:
   - `~인 상태` — 현재 상태 표현 (예: "Trade 메뉴인 상태")
   - `~된 상태` — 완료된 상태 (예: "로그인된 상태", "Limit 주문이 등록된 상태")
   - `~한 상태` — 동작 완료 (예: "약관에 동의한 상태", "Market 주문을 실행한 상태")
   - `~선택된 상태` — 선택 완료 (예: "Open Orders 탭이 선택된 상태")

### 허용 / 불허

| 구분 | 예시 |
|------|------|
| ✅ 허용 | `1. Trade 메뉴인 상태` |
| ✅ 허용 | `2. Open Order가 체크된 상태` |
| ✅ 허용 | `3. Limit Open Order 주문을 실행한 상태` |
| ✅ 허용 | `4. Open Orders 탭이 선택된 상태` |
| ❌ 불허 | `Google OAuth 로그인이 완료된 상태이다` (서술형 + "이다") |
| ❌ 불허 | `충분한 잔고가 있다` (번호 없음 + "있다" 종결) |
| ❌ 불허 | `Trade 화면에 접속해 있다` ("있다" 종결) |

### 추가 주석

Edge 조건이나 주의사항은 해당 항목 뒤에 `(* ...)` 형태로 붙인다:

```
3. 주문 수량이 부분 체결된 상태
   (* 거래량이 낮은 코인에서 체결될 가능성이 높은 호가의 size 이상 주문 후 대기하면 부분 체결됨)
```

## 9-2. 테스트 단계 / 예상 결과 작성 규칙 (마침표 / 종결)

### 원칙

**테스트 단계** 와 **예상 결과** 의 각 항목은 **문장 형식**으로 작성하므로 **반드시 마침표(`.`)로 종결**한다.

| 필드 | 형식 | 마침표 |
|------|------|--------|
| 사전 조건 | 명사형/상태형 종결 (`~인/된/한 상태`) | ❌ 불필요 |
| **테스트 단계** | **완전한 문장 (동사 종결)** | ✅ **필수** |
| **예상 결과** | **완전한 문장 (동사 종결)** | ✅ **필수** |
| 비고 | 자유 (태그 포함) | 선택 |
| 분류 / 우선순위 / 플랫폼 / 연관 화면 | 단어/코드 | ❌ 불필요 |

### 적용 대상

- 불릿(`-`) 또는 번호(`1.`) 안의 각 줄
- 한 줄 안에 여러 문장이면 **각 문장 끝**에 마침표
- 줄 내부의 `(* ...)` 보충 설명은 마지막 문자가 `)` 면 그대로 OK (마침표 추가 X)

### 예시

#### ✅ 허용

```
**테스트 단계**
1. 이메일 입력 필드에 유효한 이메일 주소를 입력한다.
2. Continue 버튼을 탭한다.

**예상 결과**
- Continue 버튼이 활성화된다.
- SCR-005 인증 코드 입력 화면으로 이동한다.
- AppState 의 `email` 값이 입력 값으로 갱신된다.
```

#### ❌ 불허

```
**테스트 단계**
1. 이메일 입력 필드에 유효한 이메일 주소를 입력한다  ← 마침표 누락
2. Continue 버튼을 탭한다

**예상 결과**
- Continue 버튼이 활성화된다  ← 마침표 누락
- SCR-005 인증 코드 입력 화면으로 이동
   (← "이동" 단어 단독 — 동사 완전형 + 마침표 필요: "이동한다.")
```

### 단어 단독 항목

극히 드물지만 단어 형태(테이블 셀, 코드 라벨 등)로 끝나면 마침표 불필요. 단 위 두 필드(테스트 단계 / 예상 결과)는 **항상 문장**으로 작성하므로 예외 없이 마침표를 붙인다.

## 12. 플랫폼별 테스트 기준

> tc-writer는 기능 TC 외에 아래 플랫폼 특화 체크리스트를 확인하고, 해당하는 TC를 MOBL 도메인 또는 관련 도메인에 추가한다.
> Phase의 플랫폼 구성은 `tc-rules-override.md`에서 확인한다.

### 12-1. 모바일 웹 공통 (Web(Mobile))

모바일 브라우저 기반 서비스라면 아래 항목을 반드시 검토한다.

| 체크 항목 | 검증 내용 | TC 추가 조건 |
|----------|----------|------------|
| **고정폭 레이아웃** | 360px(또는 Phase별 기준폭)에서 요소 잘림·겹침·가로 스크롤 없음 | 항상 추가 |
| **터치 영역** | 주요 버튼의 탭 영역이 충분히 크고, 인접 버튼 오탭 없음 | 버튼 밀집 화면에 추가 |
| **소프트 키보드** | 입력 필드 포커스 시 키보드 팝업 → 필드가 키보드에 가려지지 않음 | 입력 필드 있는 화면에 추가 |
| **바텀시트/모달** | 바텀시트·모달이 화면 전체를 덮지 않고 스크롤 가능 | 바텀시트 사용 화면에 추가 |
| **BottomNav 고정** | 스크롤 시 BottomNav가 화면 하단에 고정 유지됨 | BottomNav 있는 Phase에 추가 |
| **가로 모드** | landscape 회전 시 레이아웃 무너짐 없음 | 필요 시 추가 (Phase별 판단) |

### 12-2. iOS Safari 특화

iOS Safari에서만 발생하는 동작을 검증한다.

| 체크 항목 | 검증 내용 | 플랫폼 표기 |
|----------|----------|-----------|
| **주소창 스크롤 숨김** | 스크롤 시 Safari 주소창이 숨겨지면서 viewport 높이 변화 → BottomNav 위치 밀림 없음 | `iOS Safari` |
| **Safe Area (홈 바)** | 홈 바 영역에 BottomNav·버튼이 겹치지 않음 | `iOS Safari` |
| **노치 영역** | 노치 기기에서 헤더 콘텐츠가 잘리지 않음 | `iOS Safari` |
| **모멘텀 스크롤** | 빠른 스크롤(rubber band) 후 오버스크롤 시 UI 깨짐 없음 | `iOS Safari` |
| **키보드 팝업 처리** | iOS 키보드 팝업 시 화면이 위로 밀리거나 BottomNav가 키보드 위로 올라오지 않음 | `iOS Safari` |

### 12-3. Android Chrome 특화

Android Chrome에서만 발생하는 동작을 검증한다.

| 체크 항목 | 검증 내용 | 플랫폼 표기 |
|----------|----------|-----------|
| **뒤로가기 제스처** | Android 뒤로가기(스와이프 또는 시스템 버튼) 시 의도한 화면으로 이동 | `Android Chrome` |
| **주소창 크기 변화** | 스크롤 중 Chrome 주소창 축소·확대 시 레이아웃 재조정 정상 | `Android Chrome` |
| **키보드 팝업 처리** | Android 키보드 팝업 시 화면 리사이즈 발생 → 레이아웃 깨짐 없음 | `Android Chrome` |

### 12-4. 플랫폼별 TC 작성 원칙

1. **공통 기능 TC**는 플랫폼 필드를 `Web(Mobile)`로 표기한다.
2. **iOS 전용 동작**은 동일 TC를 `iOS Safari`로 표기하거나 별도 TC를 추가한다.
3. **Android 전용 동작**은 `Android Chrome`으로 표기한다.
4. **플랫폼 무관 로직**(서버 응답, 데이터 계산)은 `공통`으로 표기한다.
5. 플랫폼 특화 TC는 **MOBL 도메인**에 배치한다. 연관 기능에 직접 붙이는 것도 허용한다.
6. 최소 TC 세트에는 각 플랫폼별 대표 케이스를 1개 이상 포함한다.

### 12-5. 웹(Desktop) 서비스 추가 시 체크리스트

향후 데스크톱 웹 Phase가 추가될 경우 아래 항목을 검토한다.

| 체크 항목 | 검증 내용 |
|----------|----------|
| **반응형 브레이크포인트** | 768px / 1024px / 1280px 등 기준 해상도에서 레이아웃 확인 |
| **마우스 호버 상태** | hover 시 버튼·링크 색상·커서 변화 확인 |
| **키보드 단축키** | Tab 이동, Enter 실행, Esc 닫기 등 키보드 접근성 |
| **브라우저 호환** | Chrome / Safari / Firefox / Edge 별 동작 차이 |
| **우클릭·드래그** | 의도치 않은 컨텍스트 메뉴, 드래그 셀렉션 처리 |
