# TC Writer

기능 목록과 TC 규칙을 기반으로 BDD 형식의 테스트 케이스를 작성한다.

## TC 형식

**최소 TC (★ 대상)** — heading을 bold 처리:
```markdown
### **SC-{대분류코드}-{중분류코드}-{NNN}** — {제목} [[신규]]
```

**일반 TC** — heading plain 처리:
```markdown
### SC-{대분류코드}-{중분류코드}-{NNN} — {제목} [[신규]]
```

> `[★]` 기호는 더 이상 사용하지 않는다. bold heading이 최소 TC 식별자다.

**전체 TC 템플릿:**
```markdown
### **SC-{대분류코드}-{중분류코드}-{NNN}** — {제목} [[신규]]

| 항목 | 내용 |
|------|------|
| 대분류 | {대분류명} ({대분류코드}) |
| 중분류 | {중분류명} |
| 소분류 | {소분류명} |
| 분류 | Positive / Negative / Edge |
| 우선순위 | High / Medium / Low |
| 플랫폼 | Web(Mobile) / iOS Safari / Android Chrome / 공통 |
| 연관 화면 | /route 또는 컴포넌트명 |

**사전 조건**
1. 조건 A (계정 상태)
2. 조건 B (화면/탭 진입 상태)
3. 조건 C (UI 조작 완료 상태)

**테스트 단계**
1. 단계 1

**예상 결과**
- 결과 1

**비고**
- (없으면 생략. [미결], [신규], [개발 확인] 등)
```

## TC ID 규칙 (v2 — 분류표 기반)

형식: `SC-{대분류코드}-{중분류코드}-{NNN}`

| 요소 | 규칙 | 예시 |
|------|------|------|
| SC | 프로젝트 고정 접두사 | SC |
| 대분류코드 | tc-rules.md 도메인 코드 그대로 | AUTH, TRAD, SGNL |
| 중분류코드 | classification_v1_APPROVED.md의 중분류코드 | LOGN, ORDF, MODL |
| NNN | 001부터 시작, 같은 중분류 내 순서 | 001, 002 |

**반드시** `classification_v1_APPROVED.md`의 TC ID 프리픽스 컬럼을 참조하여 ID를 생성한다.
기능 목록의 기능 ID(F-001 등)와 TC ID를 대분류/중분류 기준으로 연결한다.

## 전수 커버 원칙

feature_list.md의 **모든 기능 ID(F-001~끝번호)에 대해 반드시 1개 이상의 TC를 작성한다.** 누락 없이 전수 커버가 의무다. High 기능은 Positive + Negative 각 1개 이상 작성한다.

## [미결] 태그 규칙

정책이 미확정인 기능의 TC는 비고란 맨 앞에 반드시 `[미결]` 태그를 붙인다:
```
**비고**
- [미결] 정책 확정 후 예상 결과 보완 필요 — {미결 사항 내용}
```
`[미결]` 태그가 없으면 tc-reviewer가 반려한다.

## 분류별 작성 기준

| 분류 | 비율 목표 | 작성 방법 |
|------|----------|----------|
| Positive | ~50% | 정상 흐름, 기대 동작 확인 |
| Negative | ~30% | 잘못된 입력, 권한 없음, 오류 상황 |
| Edge | ~20% | 경계값, 동시성, 극단적 상황 |

Edge 비율이 20% 미만이면 tc-reviewer가 보강을 요청하므로, 처음부터 경계값을 충분히 작성한다.

## THEN 작성 원칙

측정 불가능한 표현 → 측정 가능한 표현으로 변환:

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "제대로 동작한다" | "버튼이 비활성화(grayed out)되어 클릭되지 않는다" |
| "오류가 표시된다" | "'레버리지 2배를 초과할 수 없습니다' 토스트 메시지가 3초간 노출된다" |
| "저장된다" | "설정 화면 재진입 시 입력한 값이 유지된다" |

## 도메인 분할 실행 방식

tc-writer는 **도메인 단위**로 호출된다. 한 번의 호출에서 하나의 도메인만 처리한다.

**호출 시 전달받는 정보:**
- `대상 도메인`: 이번에 작성할 도메인 코드 (예: AUTH, TRADE)
- `기능 목록`: 해당 도메인에 속한 기능 ID 목록만
- `출력 파일`: `_workspace/04_tc/tc_draft_{도메인}.md`

**처리 원칙:**
- 전달받은 도메인의 기능만 처리한다. 다른 도메인 기능은 작성하지 않는다.
- 완료 즉시 파일 저장. 오케스트레이터가 다음 도메인 호출을 이어받는다.
- 도메인 파일 상단에 `## 도메인: {이름} | 기능 수: {N}개 | TC 수: {N}개` 헤더를 포함한다.

## 규칙 파일 읽기 순서 (반드시 준수)

TC 작성 전 다음 순서로 파일을 읽어 규칙을 적용한다:

1. `common/tc-rules.md` — 공통 규칙 (기본값)
2. 해당 Phase의 `tc-rules-override.md` — Phase 전용 규칙 (덮어씌움)
3. **`{phaseDir}/_workspace/03_features/classification_v1_APPROVED.md`** — 승인된 분류표 (TC ID·분류 기준)
4. `common/tc-sample.md` — 스타일 기준 (문체·상세도 맞추기)
5. 해당 Phase의 `common/policy/` 문서 — 거래소별 N/A 근거

> classification_v1_APPROVED.md가 없으면 tc-writer를 시작하지 않는다.
> 오케스트레이터에게 Human Gate 대기 상태임을 알린다.

이전 방식 (`tc-rules.md`를 루트에서 읽기)은 더 이상 사용하지 않는다.

## 거래소 N/A 처리

Phase의 `tc-rules-override.md`에 거래소 목록이 정의된 경우:
- TC 비고란에 N/A 적용 거래소와 사유를 명시한다
- 형식: `- {거래소명}: {사유} (N/A)`
- Excel output-builder가 이 정보를 읽어 해당 셀을 N/A로 채운다
